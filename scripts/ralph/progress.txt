## Codebase Patterns
- `createClient()` da `@supabase/ssr` (`createBrowserClient`) è un singleton — restituisce la stessa istanza ogni volta. Safe come dependency in useEffect.
- **Optimistic locking pattern**: Ogni `db.update(leads)` DEVE includere `version: lead.version + 1` nel set e il server action deve accettare `currentVersion` dal client. Check: `if (currentVersion !== undefined && lead.version !== currentVersion) return { success: false, error: 'CONCURRENCY_ERROR' }`. Client deve mostrare alert e refreshare la pagina su CONCURRENCY_ERROR.
- Supabase Realtime `payload.old` contiene solo PK senza `REPLICA IDENTITY FULL` — non fidarsi mai di `payload.old.<campo>` per logica business. Ricomputare dallo stato aggiornato.
- Il custom event `realtime_update` viene dispatchato da `useRealtimeNotifications` su INSERT notifiche e consumato da `KpiGdoBoard` per re-fetch KPI.
- I Server Action in `notificationActions.ts` ritornano `true` su successo.
- `RealtimeProvider` wrappa il dashboard layout e triggera `router.refresh()` su qualsiasi cambiamento leads.
- **Timezone pattern**: Per ottenere data `yyyy-MM-dd` in Europe/Rome usare `date.toLocaleDateString('en-CA', { timeZone: 'Europe/Rome' })` — pattern già usato in `pauseUtils.ts`. Mai usare `toISOString().split('T')[0]` o `format(new Date(), 'yyyy-MM-dd')` su Vercel (UTC).
- **Google Calendar OAuth2 client**: Mai usare un `oauth2Client` singleton a livello di modulo — su Vercel warm instances condividono stato e richieste concorrenti si sovrascrivono i credentials. Usare factory `createOAuth2Client()` per creare istanza nuova per ogni operazione.
- **Google Calendar refresh_token rotation**: Google ruota periodicamente i refresh_token. Quando si fa refresh, SEMPRE salvare `credentials.refresh_token` se presente, altrimenti il vecchio diventa invalido → `invalid_grant` permanente.
- `confermeKpiActions.ts`: i target settimanali sono su `users.confermeTargetTier1/Tier2`, quelli mensili = weekly * 4.
- **Timezone pattern (month range filter)**: Per filtrare lead per mese "YYYY-MM", usare `getMonthBounds(monthString)` per ottenere prima/ultima data del mese come stringhe, poi `toRomeStartOfDay()`/`toRomeEndOfDay()` per convertirle in Date UTC Rome-aware. Vedi `marketingActions.ts`.
- **Timezone pattern (date range filter)**: Per convertire una data "YYYY-MM-DD" (Rome) in Date UTC per filtri `gte`/`lte`, usare `Intl.DateTimeFormat('en-GB', { timeZone: 'Europe/Rome', timeZoneName: 'longOffset' })` per ricavare l'offset e costruire `new Date(\`${dateStr}T00:00:00${offset}\`)`. Vedi `toRomeStartOfDay()`/`toRomeEndOfDay()` in `archiveActions.ts`.
- **Timezone pattern (client-side display)**: Per mostrare date nel client usare `toLocaleString('it-IT', { dateStyle: 'short', timeStyle: 'short', timeZone: 'Europe/Rome' })`. Mai usare `toLocaleString([], ...)` senza `timeZone` — funziona solo se il browser è in Italy.
