# Ralph Progress Log ‚Äî CRM Fenice (Fase 2)
Started: 2026-04-07

## Codebase Patterns
- Optimistic locking: ogni UPDATE su leads DEVE includere `eq(leads.version, lead.version)` nel WHERE + version increment + check `.returning()` length
- Timezone: usare `toLocaleDateString('en-CA', { timeZone: 'Europe/Rome' })` per date stringa, mai `toISOString().split('T')[0]`
- Timezone client: `toLocaleString('it-IT', { timeZone: 'Europe/Rome' })` per display
- Google Calendar: factory `createOAuth2Client()` per ogni operazione, mai singleton
- Server Actions ritornano `{ success: boolean, error?: string }`
- Tailwind v4: custom theme via @theme in globals.css, colori brand: brand-orange (#FFBE82), brand-charcoal (#272626)
- Gamification XP formula: `100 * (Level ^ 1.5)` per target XP, 5 stadi evoluzione Fenice
- `assignedToId` e' immutabile dopo import CSV ‚Äî mai modificarlo
- DATI REALI: NON toccare lead GDO 114, appuntamenti Conferme, account esistenti

- ConfermeDrawer salesperson init: use `salespersonUserId` (UUID) NOT `salespersonAssigned` (display name) to prevent re-save corruption
- FreeBusy check: always run BEFORE db.update(), never after ‚Äî returning success:false after commit causes phantom state

## 2026-04-07 - F2-001
- **Bug**: Confirmed appointments not visible in venditori dashboard
- **Root cause (DB audit)**: ALL confirmed leads had `appointmentDate = null` (rescheduled/test data). The `isNotNull(appointmentDate)` filter in `getVenditoreAppointments` excluded them all.
- **Secondary bug**: ConfermeDrawer initialized `salesperson` from `lead.salespersonAssigned` (display name) instead of `lead.salespersonUserId` (UUID), causing data corruption on re-saves
- **Third bug**: `setConfermeOutcome` ran FreeBusy check AFTER DB update ‚Äî if check failed, DB was committed but function returned `success: false`
- **Files modified:**
  - `src/app/actions/venditoreActions.ts` ‚Äî removed `isNotNull(appointmentDate)` filter
  - `src/components/ConfermeDrawer.tsx` ‚Äî init salesperson from `salespersonUserId`
  - `src/app/actions/confermeActions.ts` ‚Äî moved FreeBusy check before DB update
- **Learnings per iterazioni future:**
  - Always check actual DB data via Supabase MCP before assuming code-level bugs
  - `salespersonUserId` is only set for confirmed leads, so it's sufficient as the sole filter
  - Never perform side-effect checks (calendar, external APIs) after DB commits
